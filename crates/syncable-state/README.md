# Syncable-State: Architecture & Internals Guide

`syncable-state` is a high-performance typed state synchronization library built on the **Authoritative Writer v1** model.

This guide is intended for developers who want to understand the deep architecture of the library: how it captures changes, tracks sequences, generates deltas, and validates synchronization paths.

---

## 1. How It Detects Changes (Explicit vs. Magic)

Unlike frameworks that use `Proxy` objects (in JS) or deep struct diffing to "magically" detect changes after the fact, `syncable-state` relies on **explicit mutable containers** (e.g., `SyncableMap`, `SyncableString`, `SyncableCounter`).

### Implicit Event Tracking

Changes are not detected passively; they are **captured actively** via implicit event tracking. When you instantiate a `RuntimeState`, it binds a shared `EventTracker` context to the state tree. Mutating the state is direct and ergonomic:

```rust
// Direct property mutation!
runtime.title.set("Hello World")?;
runtime.views.increment(1)?;
```

Every `Syncable*` container internally holds a reference to the `EventTracker` (an `Rc<RefCell<Vec<ChangeEnvelope>>>`). Inside the container's method (e.g., `increment`), the container:

1. Performs the local memory mutation.
2. Constructs a `ChangeEnvelope` representing the exact hierarchical `SyncPath` and the specific `ChangeOp` (e.g., `CounterOp::Increment(1)`).
3. Pushes this envelope into the implicit `EventTracker`.

This means **zero diffing overhead**. The library knows exactly what changed because the containers explicitly record their operations.

---

## 2. Direct Mutation & State Polling

Because we use implicit tracking, mutations are immediately applied to the local memory tree. Once you're done grouping your updates, you extract the batch by calling `poll`:

```rust
if let Some(batch) = runtime.poll() {
    // Broadcast the DeltaBatch network payload safely to clients!
    broadcast(batch);
}
```

The implicit tracker solves the topological mapping problem and ensures chronological preservation of all operations, matching exactly what the developer executed in sequence.

---

## 3. How It Batches & Compacts Changes

### Batch Generation

When you call `runtime.poll()`, the engine takes all the isolated `ChangeEnvelope`s accumulated in the implicit tracker and bundles them into a single `DeltaBatch`.

Each `DeltaBatch` is given a unique, strictly monotonic `seq` (sequence number) generated by the `ChangeCtx`.

### Compaction (v1 Behavior)

In v1, **compaction is handled entirely at the batch boundary.** The library does _not_ automatically squash multiple incremental operations deep inside a single batch (e.g., calling `increment(1)` five times simply pushes five `ChangeEnvelope`s). This intentionally keeps the logic incredibly fast and pure for chunk 1, avoiding heavy runtime introspection or AST merging.

For long-term compaction over the network, `syncable-state` relies on **Snapshots**. A peer can request a full `SnapshotBundle` (the exact materialized state at sequence `N`) instead of downloading all historical `DeltaBatch`es from sequence `0` to `N`. This completely collapses the change tree.

---

## 4. The Authoritative Writer v1 Model

`syncable-state` does not attempt to solve free-for-all conflict resolution (like pure P2P CRDTs might). Instead, it uses a highly scalable **Authoritative Writer** architecture.

### Sequence Allocation

Only **one authority** (usually the creator of the room or the central server) is allowed to mint new `seq` numbers.

- The authoritative node starts at `seq = 0`.
- Every local `commit` advances the state to `seq + 1`.

### Deterministic Remote Apply

When a non-authoritative client receives a `DeltaBatch` from the network, it validates the batch using strict continuity rules through `apply_remote`:

1. **Gap Detection:** The client expects `batch.from_seq` to match its current local `seq`. If `from_seq > current_seq`, the state is missing deltas and will reject the batch, forcing the transport layer to fetch the missing gap.
2. **Replay Rejection:** If the client receives a batch it has already applied (`from_seq < current_seq`), it silently drops it (or throws a `StaleBatch` error).
3. **Application:** If the sequence matches, the client iterates over the `ChangeEnvelope`s, traversing the memory tree via `SyncPath` and applying the `ChangeOp`. Because all peers run the exact same `ChangeOp`s in the exact same `seq` order, they all arrive at the exact same physical byte state.

---

## 5. Schema & Paths (`#[derive(SyncableState)]`)

To route a `ChangeEnvelope` to the correct deeply-nested object, the system uses a `SyncPath`.
A `SyncPath` is an array of typed path segments, such as:
`[ Field("docs"), Key("doc-123"), Field("title") ]`

The `#[derive(SyncableState)]` macro automatically generates the routing logic required to traverse a struct. When `apply_remote` is called, the macro-generated router steps through the path segments, delegating the `ChangeOp` down to the exact child container (like a `SyncableString`), which executes the final mutation.

This typed routing enforces that malformed patches from the network immediately error out (e.g., trying to route a `StringOp` to a `SyncableCounter`), preventing bad data from corrupting the memory tree.
